<p>When we create a project in Unity, Unity groups all the scripts into a single Assembly predefined by <strong><em>Unity</em></strong> as <em>Assembly-CSharp.dll</em>. This allows all of our scripts to have direct access to each other, which makes prototyping and implementation of functionality in Unity very easy.</p>
<p>However, by making parts of our code entirely dependent on each other, we risk creating spaghetti code, especially as the project grows.</p>
<p>To remove this risk of falling into a big bowl of spaghetti code, we need to remove the mono behaviours from the program logic. We can achieve this by creating our own Assembly definitions and references.</p>
<h2 id="splitting-into-assemblies">Splitting into assemblies</h2>
<p>In my unity projects, I have a <strong>Core</strong> assembly which contains all the program logic. The classes in this assembly do not inherit from <code>Monobehaviour</code> or <code>ScriptableObject</code>. This is where the meat of the code lives.</p>
<p><img src="/assets/images/posts/organizing-unity-projects-make-them-testable/cover.png" alt="Project structure" /></p>
<p>3 other assemblies depend on Core</p>
<ul>
<li><strong>Scripts</strong> contains all the mono behaviours and scriptable objects. The classes expose the properties that a designer might want to edit. They pass those values to the appropriate functions in <strong>Core</strong></li>
<li><strong>Editor</strong> contains all our custom inspectors and Editors. It only supports the Editor platform</li>
<li><strong>Tests</strong> contains all our tests. Since the <strong>Scripts</strong> and <strong>Editor</strong> do not contain any logic, we can write unit tests against the classes in <strong>Core</strong> and make sure they are modular.</li>
</ul>
<p>Besides making our code more organized and testable, splitting our code into assembly definitions helps compile times. With a single assembly definition, every time we make a change to any script, Unity has to compile every single script. By moving our code into assemblies, Unity only has to recompile the scripts that are only inside that <strong><em>Assembly</em></strong>.</p>
<p>As each assembly becomes more extensive, I split it into smaller assemblies.</p>
<h2 id="conclusion">Conclusion</h2>
<p>Unity, by default, puts all the scripts in a single assembly. This might be fine for small projects. But, by putting in a little effort and splitting our code into multiple assemblies, we can make our code more organized and testable.</p>
