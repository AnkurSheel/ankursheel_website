<p>Unity codebases have a way of turning into spaghetti. Every system talks to every other system through singletons and magic strings, and it never gets better with time. When I came back to Unity to build WordXplorer, I knew what I wanted: clean dependency injection, testable classes, no hidden dependencies. The problem was that while I'd used DI in .NET Core apps, I'd never seen it in a Unity project.</p>
<p>By the end of this, you'll have a setup where plain C# classes get proper constructor injection, MonoBehaviours register themselves automatically, and the whole thing initialises lazily without a bootstrapper. It ends up feeling a lot like any other .NET Core app which is exactly the point.</p>
<h2 id="the-alternatives-and-why-they-fall-short">The Alternatives (And Why They Fall Short)</h2>
<p>Before getting into the solution, it’s worth looking at what most Unity developers do instead. Spoilers: it’s not great.</p>
<h3 id="drag-and-drop-in-the-editor">Drag-and-Drop in the Editor</h3>
<p>The simplest way to connect objects is to expose a private serialised field and drag the reference in the Inspector. This is convenient, especially for artists and designers who can wire up behaviours without touching code. But it's a lot of manual effort and doesn't scale. And none of it is easily testable because you can't run a unit test against a scene.</p>
<h3 id="querying-the-scene">Querying the Scene</h3>
<p>You can find objects at runtime in a few ways: by name, by tag, or by type.</p>
<p>Finding by name or tag relies on magic strings. Rename something in the hierarchy and the code breaks. The worst part is you only find out at runtime.</p>
<p>Finding by type is better because it's at least type-safe. But it's still a global scene query, it's slow, and it gives you no way to swap in a different implementation for testing.</p>
<h3 id="singletons">Singletons</h3>
<p>Most Unity tutorials reach for singletons, and they are better than the two options above. But in practice, singletons cause three problems that compound as the project grows.</p>
<p><strong>Global state</strong>: Every singleton is effectively a global variable. Any class can reach in and modify it at any time, from anywhere. Reasoning about what changed, and when, becomes increasingly difficult. You can manage this if you're diligent — but diligence has a way of slipping when you're in a rush and just need to ship.</p>
<p><strong>Hidden dependencies</strong>: When a class accesses a singleton directly, that dependency is invisible from the outside. You can't tell what a class needs just by looking at its constructor or its interface. The only way to find out is to read the implementation.</p>
<p><strong>(Un)testability</strong>: Singeltons make writing automated tests hard. If a class depends on a concrete singleton, you can't swap it out for a fake in a test. You could test with the real thing, but only if it has no side effects and doesn't depend on other singletons. In practice, it usually does.</p>
<h3 id="unity-specific-di-frameworks">Unity-Specific DI Frameworks</h3>
<p>There are frameworks built specifically for Unity DI like <a href="https://github.com/modesttree/Zenject">Zenject</a> and <a href="https://vcontainer.hadashikick.jp/">VContainer</a>. Both are heavy frameworks with their own learning curve. I find them too bloated for my purposes, but they might be a viable option.</p>
<p>To be honest, I really didnt look too much into them. I might just be reinventing the wheel here.</p>
<h2 id="what-is-dependency-injection">What is Dependency Injection?</h2>
<p>DI (Dependency Injection) is a technique in OOP where instead of a class creating its own dependencies, you pass them in from the outside.</p>
<p>So instead of this:</p>
<pre><code class="language-csharp">public class GameManager
{
    private AudioManager _audio = new AudioManager();
}
</code></pre>
<p>You do this:</p>
<pre><code class="language-csharp">public class GameManager
{
    private IAudioManager _audio;

    public GameManager(IAudioManager audio)
    {
        _audio = audio;
    }
}
</code></pre>
<p>Now <code>GameManager</code> doesn’t care if it gets a real <code>AudioManager</code> or a mock for testing. A DI container automates the wiring and  resolves them wherever they're needed.</p>
<h2 id="why-does-unity-make-di-awkward">Why Does Unity Make DI Awkward?</h2>
<p>Two things get in the way in Unity.</p>
<p>First, MonoBehaviours can't be newed up. Unity owns their lifecycle by calling <code>Awake</code>, <code>Start</code>, <code>Update</code>, and so on. You can't pass dependencies through a constructor the normal way.</p>
<p>Second, there's no single entry point. A .NET console app has <code>Main</code>. An ASP.NET app has <code>Program.cs</code>. But Unity has scenes, objects, and an initialization order you don't fully control. Scenes can also be loaded and unloaded, which means objects might not always be present when the container is first created, or might be destroyed after they've already been registered.</p>
<p>You might think these two constraints would be a showstopper. They're not. But they do require a slightly different approach.</p>
<h2 id="the-solution">The Solution</h2>
<h3 id="step-1-add-the-nuget-packages">Step 1: Add the NuGet Packages</h3>
<p>Download the following DLLs from <a href="https://www.nuget.org/packages/Microsoft.Extensions.DependencyInjection/">NuGet</a> and drop them into your <code>Assets/Plugins</code> folder:</p>
<ul>
<li><code>Microsoft.Extensions.DependencyInjection.dll</code></li>
<li><code>Microsoft.Extensions.DependencyInjection.Abstractions.dll</code></li>
</ul>
<p>I've written about <a href="https://www.ankursheel.com/blog/installing-nuget-packages-unity-2021">how to install NuGet packages in Unity</a> in more detail. If you use assembly definitions, add references to these DLLs in the relevant assembly definition file.</p>
<h3 id="step-2-the-dependencyhelper">Step 2: The DependencyHelper</h3>
<p><code>DependencyHelper</code> owns the DI container and acts as the resolution point for both plain C# services and MonoBehaviours.</p>
<pre><code class="language-csharp">public sealed class DependencyHelper
{
    private static readonly Dictionary&lt;Type, object&gt; _injectedMonoBehaviours = new();
    private static readonly Lazy&lt;DependencyHelper&gt; _lazy =
        new(() =&gt; _instance ?? new DependencyHelper());
    private static DependencyHelper _instance;
    private static ServiceProvider _serviceProvider;

    private DependencyHelper()
    {
        var serviceCollection = new ServiceCollection();
        ServiceRegistry.RegisterServices(serviceCollection);
        _serviceProvider = serviceCollection.BuildServiceProvider();
        _instance = this;
    }

    public static void Register(object instance)
    {
        _injectedMonoBehaviours[instance.GetType()] = instance;
    }

    public static T GetInjectedMonoBehaviour&lt;T&gt;() where T : class
        =&gt; _injectedMonoBehaviours[typeof(T)] as T;

    public static T GetRequiredService&lt;T&gt;() where T : class
    {
        _ = _lazy.Value; // ensures the container is built on first call
        return _serviceProvider.GetRequiredService&lt;T&gt;();
    }
}
</code></pre>
<ul>
<li><code>Register</code>: called automatically by <code>InjectableMonoBehaviour.Awake</code> to store the live MonoBehaviour instance.</li>
<li><code>GetInjectedMonoBehaviour</code>: used inside <code>ServiceRegistry</code> to wire a MonoBehaviour into the container by fetching the stored instance.</li>
<li><code>GetRequiredService</code>: how everything else resolves dependencies. Throws if the service isn't registered.</li>
<li>The <code>Lazy</code> factory ensures the container is built exactly once, on the first call to <code>GetRequiredService</code>. The <code>_instance</code> check means there is only ever one container instance.</li>
</ul>
<h3 id="step-3-inherit-from-injectablemonobehaviour">Step 3: Inherit from InjectableMonoBehaviour</h3>
<p>Next, we need a way to add MonoBehaviours to the DI container</p>
<p><code>InjectableMonoBehaviour</code> is a base class that MonoBehaviours inherit from, which registers them with the container when they wake up.</p>
<p>Any MonoBehaviour that needs to be resolvable by the container should extend this:</p>
<pre><code class="language-csharp">public class InjectableMonoBehaviour : MonoBehaviour
{
    protected virtual void Awake()
    {
        DependencyHelper.Register(this);
    }
}
</code></pre>
<p>When Unity calls <code>Awake</code>, the MonoBehaviour registers itself by type. That's it. The container can now hand out a reference to this instance when something asks for it.</p>
<h3 id="step-4-the-service-registry">Step 4: The Service Registry</h3>
<p>This is where you tell the container what goes where.</p>
<pre><code class="language-csharp">public static class ServiceRegistry
{
    public static void RegisterServices(IServiceCollection serviceCollection)
    {
        // Plain C# service
        serviceCollection.AddTransient&lt;ILevelSaveDataService, LevelSaveDataService&gt;();

        // MonoBehaviour
        serviceCollection.AddTransient&lt;IAudioManager&gt;(
            _ =&gt; DependencyHelper.GetInjectedMonoBehaviour&lt;AudioManager&gt;());
    }
}
</code></pre>
<p>Plain C# services register normally. MonoBehaviours are different - they use a lambda to defer the lookup until the instance is actually needed, by which point Unity will have called <code>Awake</code> on the MonoBehaviour and registered it.</p>
<h3 id="step-5-watch-the-lifecycle-order">Step 5: Watch the Lifecycle Order</h3>
<p>This is the part that will bite you if you're not paying attention.</p>
<p>Unity calls <code>Awake</code> on scripts in an arbitrary order. If you try to resolve a dependency in <code>Awake</code>, you might run before that dependency has even registered itself. If that happens, you will get a <code>KeyNotFoundException</code> at runtime.</p>
<p>We know that <code>Start</code> is called after <code>Awake</code>, so the rule is simple:</p>
<blockquote>
<p>Always resolve and use dependencies in <code>Start</code>, not <code>Awake</code>.</p>
</blockquote>
<p>This also has the added benefit of resolving the dependency only once.</p>
<pre><code class="language-csharp">public class LevelSaveDataManager : MonoBehaviour
{
    private ILevelSaveDataService _levelSaveDataService;

    private void Start()
    {
        _levelSaveDataService = DependencyHelper.GetRequiredService&lt;ILevelSaveDataService&gt;();
    }
}
</code></pre>
<p>The same caveat applies across scenes. If a scene transition destroys and recreates MonoBehaviours, they re-register themselves in their new <code>Awake</code>. The container's service registrations stay intact - those lambdas always pull the current live instance. But if you're holding a cached reference somewhere and the MonoBehaviour was destroyed, that reference is stale. Be deliberate about when you resolve.</p>
<h2 id="isnt-this-the-service-locator-anti-pattern">Isn't This the Service Locator Anti-Pattern?</h2>
<p>Yes. I know. I'm not thrilled about it either.</p>
<p>The criticism is legitimate: when a class calls <code>DependencyHelper.GetRequiredService&lt;T&gt;()</code> directly, its dependencies are hidden. You can't tell from the constructor what it needs. Testing becomes harder because you have to know to configure the locator before running any test.</p>
<p>But Unity’s lifecycle leaves us with very few &quot;clean&quot; options.</p>
<p>Constructor injection requires that you control object creation.</p>
<p>In Unity, you don't. MonoBehaviours are created by the engine, attached to GameObjects, and initialised through <code>Awake</code> and <code>Start</code>. There's no constructor call you can intercept to inject dependencies the clean way.</p>
<p>I think the Service Locator is a small price to pay given all the other alternatives. We can minimize the damage by treating MonoBehaviours as a &quot;thin&quot; layer. They handle the Unity stuff (input, rendering), and delegate all the actual logic to injected plain C# classes.</p>
<p>As you saw in Step 5, <code>LevelSaveDataManager</code> is the thin MonoBehaviour — it knows <em>when</em> to load. <code>ILevelSaveDataService</code> is the plain C# class where all the loading logic lives, fully testable without a running scene.</p>
<h2 id="was-it-worth-it">Was It Worth It?</h2>
<p>If you've worked on .NET Core apps, you already know most of what's here. Unity just requires a small amount of extra wiring to account for its lifecycle. Once that's in place, you can mostly forget it's there.</p>
